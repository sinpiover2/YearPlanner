# Architecture Document Standards

## Purpose

This document defines standards for architecture documentation itself — not for implementation.

It exists so that architecture documents remain trustworthy as a canonical model, independent of who last edited them or which sprint produced the change. For the process used to change these documents, see the Architecture Reconciliation Workflow in `docs/WORKFLOW/DEVELOPMENT_WORKFLOW.md`.

---

## What Architecture Documents Are

- Architecture documents describe the canonical model — the responsibilities, boundaries, and information ownership of the system as it is intended to work.
- They are read before implementation, not generated by it. A document should be true regardless of which sprint is currently running.

## What Architecture Documents Are Not

- They are not implementation logs, sprint handoffs, or task lists.
- Temporary implementation details — file names, sprint numbers, in-progress caveats — belong in sprint handoffs or code comments, not in architecture documents.

---

## Principles

- **Consistent terminology.** A concept has one name. If a document introduces a replacement term, it states what the term deprecates and points to the reconciliation that made the change.
- **Distinguish architecture from implementation.** State what a subsystem is responsible for, not how the current code happens to implement it.
- **Internal consistency.** A document should not contradict itself or restate — with drift — what another document already defines.
- **Cross-reference, don't duplicate.** When a concept is owned by another document, link to it instead of restating it. See `ARCHITECTURE_INDEX.md` for the current map of ownership.
- **Reconcile before editing.** Architecture changes are analyzed against the canonical documents before any edit is made. See the Architecture Reconciliation Workflow in `DEVELOPMENT_WORKFLOW.md`.
- **Review before implementation.** Proposed architecture changes are reviewed and approved before an implementation prompt is generated.
- **QA after implementation.** Once changes are applied, the document is checked for internal consistency before it is committed.
